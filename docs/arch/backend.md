## oRPC — API Layer

oRPC is the **sole** API communication layer between frontend and backend (ADR-005).

### Key Rules

1. **No REST endpoints, no GraphQL.** All frontend-to-backend communication goes through oRPC procedures.
2. **Zod schemas define all procedure inputs and outputs.** These schemas live exclusively in `@cv-tool/contracts`.
3. **The backend implements the oRPC router.** Procedure handlers are registered in `apps/backend/`.
4. **The frontend imports the router type from `@cv-tool/contracts`** to create a type-safe oRPC client — no code generation step required.
5. **Transport is HTTP (JSON).** The backend exposes a single HTTP endpoint that the oRPC handler processes.

### Adding a New Procedure

1. Define Zod input/output schemas in `@cv-tool/contracts`.
2. Implement the handler in `apps/backend/`.
3. Call the procedure from `apps/frontend/` — type errors surface immediately if any step is inconsistent.

### OpenAPI

An OpenAPI specification is automatically generated by oRPC. It can be used for documentation or future non-TypeScript clients.

---

## Shared Type Contracts — `@cv-tool/contracts`

The oRPC router type and all Zod schemas for procedure inputs/outputs live **exclusively** in `packages/contracts/`. The backend implements the router; the frontend imports the router type.

```
@cv-tool/contracts  ←── imported by ──→  @cv-tool/frontend
                    ←── imported by ──→  @cv-tool/backend
```

Neither app defines its own copy of a shared schema.

---

## Database Access

- Only `apps/backend/` connects to PostgreSQL. The frontend **never** accesses the database directly.
- All data access is mediated through oRPC procedures.
- See [data-model.md](./data-model.md) for database schema, migrations, and connection details.
